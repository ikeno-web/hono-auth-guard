# 要件定義書: hono-auth-guard

## 1. アプリ概要

hono-auth-guard は Cloudflare Workers + Hono 向けの軽量認証ガードライブラリである。
JWT検証、APIキー認証、OAuthプロバイダトークン検証を統合的に扱い、型安全なコンテキストを提供する。
「Guard（ガード）」とは認証条件を宣言的に定義するミドルウェア生成関数であり、本ライブラリの中心概念である（例: `JwtGuard`, `ApiKeyGuard`）。
Hono の peer dependency 以外の外部依存をゼロとし、全暗号処理に Web Crypto API を使用する。
Workers のCPU制限（free: 10ms / paid: 50ms）内で動作するよう設計し、バンドルサイズ10KB未満を目標とする。

## 2. ターゲットユーザーペルソナ

### ペルソナA: インディー開発者 — SaaS APIビルダー
- **属性**: 個人〜2-3人チーム。Workers + Hono でSaaS APIを構築中
- **技術水準**: TypeScript中級以上。Hono経験あり。認証は自前実装経験が少ない
- **課題**: JWT検証を毎回手書きしている。API Key管理も場当たり的。統合的なソリューションが欲しい
- **期待**: `npm install` 一発で認証が組み込める。型安全。ドキュメントが充実
- **利用場面**: 新規APIのプロトタイピング〜本番運用

### ペルソナB: チーム — Express→Hono移行組
- **属性**: 5-15人の開発チーム。Express + passport.js から Hono + Workers へ移行中
- **技術水準**: TypeScript上級。passport.jsのようなミドルウェアパターンに慣れている
- **課題**: passport.js相当の認証ミドルウェアがHono/Workers向けに存在しない。移行コストを最小化したい
- **期待**: 既存のJWTやAPIキー戦略をそのまま載せ替えられる。カスタマイズ性が高い。テスト容易
- **利用場面**: 既存サービスのWorkers移行プロジェクト

## 3. ユーザーストーリー

| ID | ストーリー |
|----|-----------|
| US-01 | API開発者として、JWTトークン（HS256）をリクエストヘッダーから検証したい、なぜなら認証済みユーザーだけにAPIアクセスを許可するため |
| US-02 | API開発者として、RS256署名のJWTを検証したい、なぜなら非対称鍵による安全な認証を行うため |
| US-03 | API開発者として、APIキーをカスタムヘッダー（X-API-Key等）から取得・検証したい（ApiKeyGuard）、なぜならサービス間通信でAPIキー認証を使うため |
| US-04 | API開発者として、複数のGuardをルートごとに使い分けたい、なぜなら公開API・認証済みAPI・管理者APIで異なる認証が必要なため |
| US-05 | API開発者として、認証失敗時にRFC 7807準拠のエラーレスポンスを返したい、なぜならクライアント側で一貫したエラーハンドリングを実装するため |
| US-06 | API開発者として、JWTのカスタムクレーム（role, org_id等）を型安全に取得したい、なぜなら後続のハンドラーでユーザー情報を安全に参照するため |
| US-07 | API開発者として、ロールベースのアクセス制御（admin/user等）をGuardで宣言的に記述したい、なぜなら認可ロジックをハンドラーから分離するため |
| US-08 | API開発者として、GoogleのOAuthトークンを検証したい、なぜならGoogleログインしたユーザーのAPIリクエストを認証するため |
| US-09 | API開発者として、GitHubのOAuthトークンを検証したい、なぜならGitHub認証ベースのAPIを提供するため |
| US-10 | API開発者として、JWKSエンドポイントから公開鍵を自動取得・キャッシュしたい、なぜなら鍵ローテーションに手動対応したくないため |
| US-11 | API開発者として、GuardをCORSミドルウェアと組み合わせて使いたい、なぜならブラウザからのAPI呼び出しでpreflight + 認証を正しく処理するため |
| US-12 | API開発者として、テスト時にモック認証コンテキスト（`createMockAuth()`等）を簡単に作成したい、なぜならユニットテストで実際のJWT生成なしにハンドラーをテストするため |
| US-13 | API開発者として、レートリミティングと認証を組み合わせたい、なぜなら認証済みユーザーごとにリクエスト制限を設けるため |
| US-14 | API開発者として、認証成功時にカスタムロジック（ログ記録、メトリクス等）を挟みたい、なぜなら認証イベントの監査証跡を残すため |

## 4. 機能一覧（MoSCoW分類）

### Must（MVP必須）

| ID | 機能 | 関連US |
|----|------|--------|
| F-01 | JwtGuard — JWT検証（HS256） | US-01 |
| F-02 | JwtGuard — JWT検証（RS256） | US-02 |
| F-03 | ApiKeyGuard — APIキーヘッダー認証 | US-03 |
| F-04 | Guard合成（ルートごとの認証方式切替） | US-04 |
| F-05 | 型安全なコンテキスト（`c.get('auth')` でクレーム取得） | US-06 |
| F-06 | RFC 7807準拠エラーレスポンス | US-05 |
| F-07 | カスタムクレームバリデーション（述語関数ベース: `(claims) => boolean`）。コアGuardに組み込みの簡易バリデーション。複雑なスキーマ検証はF-11（Zod連携）で対応 | US-06 |
| F-16 | テスト用モックヘルパー（`createMockAuth()`, `createMockContext()`等）。ライブラリ利用者がユニットテストで実際のJWT生成なしにハンドラーをテストできるユーティリティ | US-12 |

### Should（v1.x で実装予定）

| ID | 機能 | 関連US |
|----|------|--------|
| F-08 | OAuthGuard — OAuthプロバイダトークン検証（Google） | US-08 |
| F-09 | OAuthGuard — OAuthプロバイダトークン検証（GitHub） | US-09 |
| F-10 | JWKSエンドポイント自動取得・キャッシュ | US-10 |
| F-11 | カスタムクレームバリデーション（Zodスキーマ連携）。F-07（述語関数ベース）の拡張で、オプショナル依存としてZodスキーマによる型安全バリデーション + 型推論を提供する。`zod` は peer dependency として任意導入 | US-06 |
| F-17 | CORS互換性（preflightリクエスト時のGuardスキップ）。Hono標準CORSミドルウェアとの併用を保証し、OPTIONSリクエストで認証を要求しない | US-11 |

| ID | 機能 | 関連US |
|----|------|--------|
| F-12 | レートリミティング統合フック | US-13 |
| F-13 | RoleGuard — RBACヘルパー（ロール/パーミッション宣言） | US-07 |
| F-14 | リフレッシュトークンローテーション | — |

### Won't（スコープ外）

| ID | 機能 | 理由 |
|----|------|------|
| W-01 | セッション管理 | ステートレスミドルウェアの範囲を超える |

| W-03 | フロントエンドUIコンポーネント | バックエンド専用ライブラリ |
| W-04 | トークン発行（JWTの生成） | 検証専用。生成はアプリ側の責務 |
| W-05 | パスワードハッシュ・管理 | 認証プロバイダの責務 |

## 5. 非機能要件

### 5.1 性能
- JWT検証（HS256）: 1ms以内
- JWT検証（RS256）: 5ms以内
- APIキー検証: 0.5ms以内
- Workers CPU制限内での動作: free tier 10ms / paid tier 50ms

### 5.2 バンドルサイズ
- コアモジュール（JWT + API Key）: 5KB未満（minified + gzipped）
- 全機能含む: 10KB未満（minified + gzipped）

### 5.3 互換性
- Hono: v4.x 以上
- Cloudflare Workers runtime
- Miniflare（ローカル開発）
- Web Crypto API 準拠環境

### 5.4 セキュリティ
- タイミング攻撃耐性のある文字列比較
- JWTの `alg: none` 攻撃の拒否
- 期限切れトークンの確実な拒否（exp クレーム検証）
- 公開鍵アルゴリズムの強制（RS256指定時にHS256すり替えを拒否）

### 5.5 開発者体験
- TypeScript strict mode 完全対応
- IntelliSense でクレームの型補完が効くこと
- テストカバレッジ 90%以上
- ドキュメント（README + JSDoc）が英語で整備されていること

### 5.6 ライセンス
- MIT License で公開する（OSSとしての採用障壁を最小化するため）

## 6. スコープ外の明示

以下はこのプロジェクトのスコープに含めない:

- **セッション管理**: Cookie/セッションベースの認証はステートフルであり、Workers向けミドルウェアの責務を超える
- **データベース統合**: ユーザーストア、APIキーストアのDB実装はアプリ固有のため除外。バリデーション関数のインターフェースのみ提供
- **フロントエンドUI**: ログインフォーム、認証モーダル等のUIコンポーネントは含まない
- **トークン発行**: JWT生成、OAuthフローのトークン取得は含まない。検証のみ
- **パスワード管理**: パスワードハッシュ、リセットフロー等は含まない
- **マルチテナンシー**: テナント分離ロジックはアプリ層の責務
- **Node.js / Deno / Bun 互換**: Workers + Web Crypto API 専用。他ランタイムは明示的にサポート対象外（動く可能性はあるが保証しない）

## 7. 用語集

| 用語 | 定義 |
|------|------|
| Middleware | Honoのリクエスト処理パイプラインに挿入される関数。`(c, next) => ...` 形式 |
| Guard | 認証条件を宣言的に定義するミドルウェア生成関数。本ライブラリの中心概念。GuardはMiddlewareの一種であり、`JwtGuard()`, `ApiKeyGuard()`, `OAuthGuard()` 等の形式でHonoミドルウェアを生成する |
| Claim | JWTペイロードに含まれるキーバリューペア（sub, iss, exp, カスタムクレーム等） |
| Bearer Token | `Authorization: Bearer <token>` ヘッダーで送信されるトークン文字列 |
| JWKS (JSON Web Key Set) | 公開鍵の集合をJSON形式で公開するエンドポイント。RS256検証時に使用 |
| API Key | サービス間通信やサードパーティ連携で使用する静的な認証キー |
| RBAC (Role-Based Access Control) | ユーザーに付与されたロール（admin, user等）に基づくアクセス制御 |
| RFC 7807 | HTTP APIのエラーレスポンス標準形式（Problem Details） |
| Web Crypto API | W3C標準のブラウザ/Workers向け暗号API。`crypto.subtle.*` |
| Tree-shaking | バンドラーが未使用コードを除去する最適化手法 |
| Timing-safe comparison | 実行時間から秘密情報を推測するタイミング攻撃を防ぐ定数時間比較 |
| Algorithm substitution attack | JWTの `alg` フィールドを差し替えて署名検証を回避する攻撃手法 |
